Storefront · internal-tools scaffold

The Scaffold Monorepo

One lookup source, per-tool repos out. A requester's problem statement lands in one category, which picks one preset, which decides exactly what repository we package and hand over. Nothing runs from this monorepo — it's the source we carve from.

01

How a request becomes a repository

The scope agent's answers flow left to right. Gate 0 decides whether a repo is handed over at all; the preset catalogue decides which one.

Input answers.json 10-question checklist filled by the scope agent
›
Gate 0 Is this even a repo? Routes non-app requests out — no repo produced
›
Lookup preset + modules Answers map to one of 11 shapes and its flags
›
Compose base + overlays Skeleton merged with preset & module files
›
Package headout/<tool> Standalone repo pushed, Guardian roles wired, auto-listed

└─ Requests that fail Gate 0 branch off here → routed to a no-code workflow, a skill, a script, or an existing tool. No repository.

02

What's in the monorepo

Four layers. The split that makes it work: the framework travels to every emitted repo as a versioned dependency (never copied), while the skeleton, presets, and modules are rendered as owned source the builder edits.

packages/
Shared framework — design system, Guardian auth recipe, BFF kit, data/async kits, observability.
→ travels as @headout/* dependency
bases/app-base/
The skeleton every emitted repo starts from — frontend shell, backend BFF entry, CI, agent skills.
→ rendered as owned source
presets/ + modules/
The 11 shapes and the composable flags (data, auth, async, integrations). Each is a manifest + overlay.
→ rendered as overlay files
scaffold/
The engine — registry (answers→preset), compose (merge overlays), package-repo (emit & register).
→ runs the lookup, never ships

"Base" is not one fixed bundle a tool either fully gets or doesn't. core ships to every repo; everything else in app-base is a slice a preset either includes or drops. A reporter gets no web UI and no HTTP listener at all — it isn't "base minus something," it never had them.

What's actually in core — nine items, every repo, regardless of interface:

File / folderWhat it is
package.jsonThe workspace manifest — declares which @headout/* framework packages this repo depends on. Guardian auth + the relevant kit are always there; umbreon is added only when a frontend-shell is also present.
tsconfig.jsonExtends @headout/tsconfig's strict base — the app/node/react variant matches whichever shells exist.
.github/workflows/ci.ymllint → typecheck → test → build, using @headout/eslint-config. The design-system rules only fire on repos that actually have a frontend-shell to lint.
.agents/skills/guardian-auth, api-integration, use-design-system — copied in so the builder's Claude Code follows the verified recipe instead of reinventing auth or downstream wiring.
AGENTS.mdEmbeds Building-a-scaled-app (PRD → prototype → system design → chunked implementation), plus this preset's specific conventions and its graduation triggers.
.env.exampleEvery env var this composed repo actually needs — Guardian config always, plus whatever each turned-on module adds (DATABASE_URL, KAFKA_BROKERS, …).
README.mdWhat the tool is and how to run it locally — plus, for mcp-server, the claude mcp add / .mcp.json connection snippet.
docker-compose.ymlLocal dev services. Present in every repo, but stays close to empty unless a data/cache module adds a service to it.
Dockerfile + deploy manifestThe deploy target — an explicit open item, not a settled default (incumbent AWS/ODE pipeline vs Railway).

core never contains business logic, models, routes, or UI components — none of that is interface-agnostic, so none of it belongs here. That's exactly what the shells, the preset overlay, and the modules add on top.

03

The preset catalogue

If the problem is this → we hand over that. Each card spells out exactly what lands in the repo: which shells from app-base it keeps (a tool never gets shells it has no use for — see the matrix above), and the real folders the preset + its modules add on top. There are 11 cards because there are 11 different problem shapes — each names what makes it distinct from its look-alikes.

Examples name the shape, not the size

Each card lists real Headout tools that share its problem shape. A couple are heavyweight engineering products — eng-grade ceiling Delphi (codebase-intelligence MCP), Plato (autonomous coding bot). Nobody self-serves a Delphi — and that's the point. Delphi solves one problem at full engineering depth, but the shape of it — "let an agent answer questions about our data" — is one a non-technical team can meet with a simple chatbot. The example marks the shape; the scaffold hands you the lightweight version.

Gate 0 — the "no repo" categories

Some problem statements get no repository, because a repo is the wrong answer. These route out before the catalogue is ever reached — only the 11 presets above produce a packaged repository.
